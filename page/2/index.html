数据库常见的索引有哈希表、有序数组、搜索树，此外还有一些跳表等，在此先不赘述。
哈希表的通过哈希函数将key的位置确定，因此查找的时间是O(1)，如果有哈希地址一样的，就拉出一个链表。缺点是不能范围查找，因此不适合一般场景，适合只有等值查询的场景，比如KV数据库。　　
有序数组在范围查找方面就很优秀,先用二分查找,时间是O(logN),找到以后,可以向右遍历,所以单看范围查询,有序数组是最好的数据结构了,但是如果要插入新的数据,后面的所有数据都要移动,因此只适用于静态数据.
二叉搜索树的查询也是O(logN),但是维护这样一个搜索树的时间复杂度也是O(logN),有时需要分裂合并造成的开销也很大.二叉树的搜索效率比多叉树要高,那么为什么要用N叉树呢?
因为索引的一部分还是在磁盘里的,读一个数据块还是要花费很长的时间,如果树高太长,要读很多次磁盘,所以尽量保持树高小.这里的N要取决于数据块的大小.
以一个InnoDB的整型字段索引为例,N差不多是1200,那么在树高是4的时候,1200^3已经是17亿多了.访问磁盘的次数就只需要三次了.所以,因为N叉树优秀的读写性能和适配磁盘的访问方式,广泛的运用在数据库引擎中.
说完索引的的不同数据结构,再来看看具体InnoDB的索引吧.

InnoDB的索引结构

InnoDB中每个索引代表一个B+树,为什么不用B树呢,大家都知道B树的每个节点都会存data,而B+树的data只存在叶子节点上,也就是说,B树的每个节点会更大,这样会增加磁盘IO次数,因为N变小了,另外一点是B+树的叶子节点是用指针串起来的,能很好地支持范围查询.
这里插播一下InnoDB和MyISAM的区别吧:

  • MyISAM不支持事务,InnoDB支持事务
  • MyISAN的锁是表极锁,InnoDB支持到行级锁
  • MyISAM支持全文类型索引,InnoDB不支持
  • MyISAM相对简单，效率上要优于InnoDB
  • 在读远远多于锁,且不需要使用事务的时候,可以选择MyISAM

在说一点结构上的区别,MyISAM的索引和数据是分开存放的,data里是数据的地址,索引放在XX.MYI文件中，数据放在XX.MYD文件中，所以也叫非聚集索引.

InnoDB的data存的就是数据本身,存在一个叫XX.IDB的文件里.也叫聚集索引.

主键索引也被称为聚簇索引,非主键索引被称作二级索引.

而如果查询select from T where k = 200,需要先查k索引树,拿到id以后再去主键索引查一遍,这个操作叫做回表.

索引维护

因为B+树需要维护,具体的算法可以参见<数据库系统概念>,总之如果数据页满了,需要分裂,或者数据页的合并,因此我们应该尽量定义自增主键.
自增主键 NOT NULL PRIMARY KEY AUTO_INCREMENT,系统会自动按自然数顺序给新的记录赋值,也就是说,每次新增记录都是追加操作,不会引起其他数据的移动,而如果用有业务逻辑的字段做主键,则一般没有顺序,写数据成本较高.
另外可以从空间角度考虑,如果有一个字符串类型是逻辑上唯一的,用它做主键是不是也很好呢?其实不是,因为非主键索引存的是主键的值,而bigint做主键也只需8个字节,而字符串类型可能会占用更多的字节.例如,用utf8编码的话,一个汉字占三个字节,英文占一个字节,空间消耗明显更大.
当然也是有场景适合用业务字段直接做主键:

  • 只有一个索引
  • 该索引必须是唯一索引

这是典型的KV场景,这时候直接将这个索引设为主键,避免了回表.

覆盖索引

如果我们执行select * from T where k between 3 and 5,需要回表两次,而如果执行select ID from T where k between 3 and 5,就只需要在k索引树上查找了,因为索引k已经覆盖了我们的查询需求.
由于覆盖索引可以减少树的搜索次数，显著提升查询性能，所以使用覆盖索引是一个常用的性能优化手段.

数据库事务是针对一串操作，保证它要么全部执行完，要么一个都不执行。有四个指标：ACID，也就是原子性、一致性、隔离性、持久性，先说说i，也就是隔离性。
在MySQL中，事务是在引擎层实现的，InnoDB有事务，而MyISAM没有，这是它被取代的重要原因之一。说道差异，见面试系列（四）吧。
当数据库有多个事务同时运行的时候，可能会出现脏读、幻读、不可重复读的问题，为了解决这些问题，有了隔离级别的概念。事务的隔离级别，主要有：

  • 读未提交：一个事务还未提交，他的变更就能被其他事务看到
  • 读提交：一个事务提交以后才能被其他事务看到
  • 可重复读：一个事务执行中看到的数据，不管其他事务有没有改变，都不会发生变化
  • 串行化：就是加锁，读锁或者写锁

实现上，在事务创建的时候会生成一个视图，访问的时候以视图的结果为准。在“读提交”下，视图是每个sql语句开始的时候创建的，“读未提交”下没有视图的概念，直接返回最新值，在“可重复读”下，视图是从事务开始就一直存在的。“串行化”根本就不会并发读一条记录。

传统的UNIX系统实现在内核中设有高速缓存区或页高速缓冲，大多数磁盘I/O都通过缓冲区进行，当我们向文件中写入数据时，内核通常先将数据复制到缓冲区中，然后排入队列，晚一些再写入磁盘，这种方式称为延迟写(delayed write)。
延迟写减少了磁盘I/O次数，但是降低了文件的更新速度，为了保证磁盘上实际文件系统和缓冲区内容的一致性，UNIX系统提供了sync、fsync、和fdatasync三个函数。

“Unfortunately fsync() will always initialize two write operations : one for the newly written data and another one in order to update the modification time stored in the inode. If the modification time is not a part of the transaction concept fdatasync() can be used to avoid unnecessary inode disk write operations.”

“fdatasync does not flush modified metadata unless that metadata is needed in order to allow a subsequent data retrieval to be corretly handled.”

1.给log文件编号，每个固定10MB，从1开始编号，名称格式为“log.%010d”
2.在第一次写log文件时，先写最后1个page，拓展为10MB大小
3.向log文件中追加记录时，由于文件的尺寸不发生变化，使用fdatasync可以大大优化写log的效率
4.如果写满了，就新建一个log文件

一般来说，一条查询语句会经过连接器、分析器、优化器、执行器，最后由执行引擎获取到结果。
mysql还有两个日志模块，redo log和binlog，mysql里经常用到Write-Ahead Logging技术，也就是先写日志，再写磁盘。redo log是有大小限制的，其实就像是一个循环队列，有两个标记，write pos和checkpoint，write pos是当前写的位置，checkpoint是当前要擦除的位置，擦除前要把记录更新到磁盘。这样可以保证即使数据库异常重启，也能保证crash-safe。
redo log包括两部分，一是缓冲redo log buffer，二是持久化的redo log file，InnoDB通过force log at commit实现事件的持久性，即在事务提交中，必须先将所有的事务日志写入磁盘的redo log file和undo log file，为了确保每次日志都能写入到事务日志文件中，在每次将log buffer中的日志写入日志文件的过程中都会调用一次操作系统的fsync操作(即fsync()系统调用)。参考面试系列第二篇。

实模式(real mode)和保护模式(protected mode)

这是CPU的工作模式，在早期CPU上运行的是实模式，而现代CPU用的是保护模式，至于为什么现代PC也要从实模式启动，是为了向后兼容。
实模式出现于早期8088CPU时期，当时硬件性能有限，只有20位地址线，地址空间只有1MB，以及8个16位的通用寄存器，以及4个16位的段寄存器，因为只有十六位，所以想指向20位的地址需要进行一些变换。

(段基址：段偏移量)，第一个字段是段基址，有段寄存器提供%cs，%ds，%ss，%es的。具体这个指令采用哪个段寄存器是由这个指令的类型来决定的。比如要取指令就是采用%cs寄存器，要读取或写入数据就是%ds寄存器，如果要对堆栈操作就是%ss寄存器. 第二个是段内偏移量，代表访问的内存距离段基址的偏移，由通用寄存器提供。这里主存的计算方式是 段基址向左偏移四位，再与段内偏移量相加。
所以假设 %cs中的值是0xff00，%ax = 0x0110。则(%cs:%ax)这个地址对应的真实物理地址是 0xff00<<4 + 0x0110 = 0xff110。
但是！现在的CPU地址线个数已经变成了32根，所以可访问内存到了4GB，寄存器也变成32位的，所以原来的内存地址计算方式不再适合了。
保护模式的内存访问方式参见CSAPP第九章虚拟内存
